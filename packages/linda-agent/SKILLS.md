# Скиллы Linda

В Linda **скилл** — это небольшой execution-модуль. Он не решает маршрут сам по себе. Он получает уже выбранную движком задачу и исполняет её в нужном тоне и с нужными ограничениями.

## Как это работает

1. **Источник правды**: `psf-engine-v2` возвращает `activeSkill`, `allowedSkills`, `nextBestAction`, `humanHandoff` и `skillContext` перед каждым ходом.
2. **Загрузка**: `LindaClientAgent` или `LindaAdminAgent` ищет `skills/<skill_id>/SKILL.md`.
3. **Исполнение**: содержимое `SKILL.md` объединяется с backend-контекстом. LLM не решает глобальный маршрут, а исполняет текущий шаг.
4. **Смена шага**: если нужен другой skill или handoff человеку, это должно быть отражено в решении движка, а не захардкожено в skill-файле.

Короткое правило:

- движок = мозг;
- `linda-agent` = исполнитель;
- skills = маленькие управляемые блоки поведения.

## Каталог скиллов

### Client skills

#### `manager`

Базовая личность клиентского агента для спокойного conversational слоя.

- **Цель**: ответы на общие вопросы, удержание интереса, общая консультация.
- **Тональность**: экспертная, теплая, ненавязчивая.
- **Когда используется**: когда движок не включил более узкий execution-skill.

#### `problem_discovery`

Диагностика запроса клиента до рекомендаций или записи.

- **Цель**: понять, с чем именно пришёл человек, чего он боится, насколько он готов двигаться дальше.
- **Что делает**:
  - помогает прояснить проблему;
  - отделяет размытый запрос от явного booking intent;
  - подготавливает переход в `profile_enrichment`, `service_recommendation`, `booking_consultation` или `human_handoff`.
- **Важно**: это не главный роутер системы. Финальное решение о `activeSkill` всё равно за движком.

#### `profile_enrichment`

Навык для квалификации пациента и добора данных профиля.

- **Цель**: собрать недостающие поля профиля, без которых движок не может безопасно и глубоко персонализировать рекомендации.
- **Принцип работы**:
  1. Движок видит, что для персональной рекомендации данных недостаточно.
  2. Возвращает `activeSkill=profile_enrichment`.
  3. В `skillContext` уже лежат `missingFields`, `questions`, `sessionId` и `enrichmentLink`.
  4. Агент либо задаёт вопросы в чате, либо мягко отправляет человека по enrichment-ссылке.
- **Важно**: ссылку на интейк генерирует не skill сам по себе, а движок.

#### `service_recommendation`

Навык глубоко персональных рекомендаций.

- **Цель**: предлагать конкретные услуги и товары клиники на основе профиля, истории переписки, прошлых процедур, страхов, бюджета и предыдущих сигналов интереса.
- **Принцип**:
  - не выдумывать услуги и товары;
  - не давать шаблонные офферы;
  - если данных не хватает, не импровизировать, а уводить в enrichment-контур.

#### `annual_plan_tracking`

Навык долгосрочного сопровождения.

- **Цель**: предложить годовой план и сопровождать его выполнение по этапам.
- **Когда нужен**: для повторяемых процедур, клубных форматов, длительных планов ухода.
- **Статус**: execution-skill уже есть в каноне `linda-agent`, но его включение в реальный runtime должно быть явно добавлено в движке.

#### `booking_consultation`

Помощь в выборе времени и процедуры.

- **Цель**: конвертация интереса в конкретную запись.
- **Тональность**: четкая, ориентированная на результат.

#### `human_handoff`

Системный escape hatch, а не "редкий спецслучай".

- **Цель**: передать человека оператору/врачу на любом этапе.
- **Когда нужен**:
  - клиент сам просит человека;
  - есть риск, раздражение, недоверие, нестандартный кейс;
  - движок рекомендует handoff;
  - вопрос выходит за рамки безопасной автоматизации.
- **Важно**: handoff должен считаться допустимым исходом любого skill flow.

### Admin skills

#### `admin_assistant`

Скилл для `LindaAdminAgent`, используемый персоналом клиники.

- **Цель**: мониторинг и управление активными сессиями клиентов.
- **Возможности**: просмотр статусов анкет, отправка сообщений "от имени" Линды, принудительное изменение полей профиля.

## Принципы проектирования

- Делайте skills маленькими. Один skill = одна чёткая задача.
- Не пытайтесь собрать "универсального продавца" в одном `SKILL.md`.
- Всё, что похоже на выбор маршрута, должно жить в движке.
- Всё, что похоже на стиль исполнения шага, может жить в skill.
- Если skill нельзя безопасно исполнить без дополнительных данных, он должен честно опираться на enrichment/handoff, а не фантазировать.

Нормальная декомпозиция:

- `problem_discovery`
- `profile_enrichment`
- `service_recommendation`
- `booking_consultation`
- `annual_plan_tracking`
- `human_handoff`

## Как добавить новый скилл

1. **Создайте директорию**: `packages/linda-agent/skills/<id_скилла>/`.
2. **Создайте файл `SKILL.md`**:
   - Опишите роль ("Что именно ты делаешь?").
   - Укажите рамки ("Чего ты не делаешь?").
   - Опишите, какой контекст skill обязан учитывать.
   - Опишите, в каких случаях нужно не продолжать автоматически.
3. **Зарегистрируйте**: добавьте ID в `LindaRuntimeConfig.clientAgent.enabledSkills`.
4. **Если skill должен реально работать в проде**: научите движок выдавать этот `activeSkill`. Без этого файл будет просто лежать в каталоге.

### Советы по написанию `SKILL.md`

- **Лаконичность**: длинные манифесты обычно только размывают поведение.
- **Однозадачность**: не смешивайте discovery, enrichment, recommendation и booking в один skill.
- **Явные стоп-условия**: пропишите, когда нужен handoff или возврат к enrichment.
- **Контекст**: если skill должен опираться на историю диалога, прошлые процедуры, бюджет или страхи, это надо прямо написать.

## Что важно про enrichment и handoff

- `profile_enrichment` не должен сам выдумывать enrichment-ссылку. Если движок выбрал этот skill, он должен прислать `enrichmentLink` в `skillContext`.
- `human_handoff` должен быть доступен на каждом этапе как системное правило.
- Если для персональной рекомендации данных мало, агент не должен "как-нибудь продать". Он должен добрать данные или передать человеку.
